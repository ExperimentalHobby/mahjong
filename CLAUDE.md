# CLAUDE.md

このファイルは、本リポジトリで作業する Claude Code (claude.ai/code) へのガイダンスを提供します。

## プロジェクト概要

**人間 vs CPU AI** の日本式リーチ麻雀ゲーム。

- **モード**: 対局開始時に **二人麻雀** / **四人麻雀** をユーザーが選択し、モードごとにルール・卓構成を切り替える
- **AI**: C# で直接実装する CPU AI（Python/Rust を介した別プロセス構成は採用しない。三層IPCはOthello_CSPY固有の構成であり本プロジェクトでは不要）
- **UI**: C# .NET 10 WPF（MVVM パターン）。WinUI3・コンソール版は現時点では対象外(将来的に追加検討の可能性あり)
- **ルール**: 一般的な日本式リーチ麻雀を採用する
  - 四人麻雀: 天鳳ルールをベースとした一般的なルール(喰いタン・後付けあり 等)
  - 二人麻雀: 代表的な二人打ちルールを採用する
  - 得点表・役一覧・ドラ表示条件・二人麻雀の牌山構成などの詳細仕様は本ドキュメントでは確定しない。各機能の実装着手時に `docs/{機能名}-Plan.md` で個別に確定させる

## アーキテクチャ方針

まだ実装が存在しないため、以下は「今後この方針で設計する」という指針であり、実装が進むにつれて本セクションを実態に合わせて更新していく。

### モード分岐（GameMode）
- `GameMode`（`TwoPlayer` / `FourPlayer` 等）を対局開始画面でユーザーが選択する
- プレイヤー人数・王牌枚数・配牌ロジック・成立しうる役やルールの差分は、モードごとの「ルールセット」オブジェクトに閉じ込め、ゲームロジック中核（`MahjongEngine` 等）はモード共通のインターフェース経由で呼び出す
- 二人麻雀と四人麻雀で異なるロジック（牌山構成、鳴きの制限など）は共有コードと分岐コードを明確に分離し、一方のモード向け変更が他方のモードを壊さないようにする（テストもモードごとに分けて用意する）

### ゲームロジック中核
- 配牌・ツモ・打牌・鳴き（ポン・チー・カン）・リーチ・和了判定・流局判定を担当するクラスを設計する
- 卓状態はミュータブルになる想定。Othello の `Board.Clone()` と同様、履歴・Undo・他オブジェクトへ渡す前に必ずクローンする設計にする
- 天和・地和など「第一巡限定」の役や、槍槓・嶺上開花・海底撈月・河底撈魚など「発生タイミング限定」の役は、状態遷移のどの時点で判定するかを設計時に明確化する

### AI（CPU戦略）
- Othello の `IAIStrategy` 注入パターンを踏襲し、モード・難易度ごとに差し替え可能なインターフェースを設計する
- 評価関数・シャンテン数計算・危険牌判定などはモード共通処理として実装し、ルール差分のみをルールセット経由で吸収する
- 難易度段階の要否や具体的な思考アルゴリズム（シャンテン数ベースの貪欲法／探索ベース 等）は、AI実装着手時に別途プランで確定する

### ViewModel / WPF UI
- Othello の `GameViewModel` と同様、AI をコンストラクタインジェクション（`Func<..., IAIStrategy>?` 等）で受け取り、テスト時は Fake に差し替え可能にする
- MVVM パターンを踏襲する
- 対局開始画面で二人麻雀/四人麻雀を選択後、対応するモードの卓画面へ遷移する構成にする

## 作業ルール

- **実装前にプランを提示すること**: コード変更を行う前に、方針・変更箇所・影響範囲を日本語でまとめたプランを提示し、ユーザーの承認を得てから実装に進む。
- **承認されたプランは `docs/{機能名}-Plan.md` として書き出す**（`docs/` は `.gitignore` 対象のローカル資料であり PR には含まれない）。実装の経緯を後から追えるようにする
- **複数項目をまとめて依頼された場合は 1 項目ずつサイクルを完結させること**: 「プラン提示 → 承認 → TDD 実装 → ビルド/テスト確認 → コミットメッセージ提示 → 承認 → コミット」のサイクルを 1 項目ごとに完了させてから次の項目に進む。TodoWrite で全項目の進捗を管理し、項目間で状態を見失わないようにする
- **軽微な修正は直接編集可**: 以下の軽微な修正はプラン提示なしで直接編集してよい。ただし変更内容が明らかな場合のみ:
  - スペルミス・タイポ修正
  - 単純な値の誤り（符号違い `+`/`-`、定数値の訂正など）
  - コメント・ドキュメントの文言修正（既存ドキュメントの改善）
  - IDE 警告削除（未使用 using、未使用フィールドなど）

  ただし、以下のような変更は軽微ではないため、プラン提示後に実装すること: ロジック変更が伴う修正、ファイル追加・削除、テスト追加、依存関係の変更
- **GitHub Flow に従うこと**: コード変更は必ず feature ブランチで行い、`main` に直接コミットしない。

  **ブランチ運用手順（GitHub Flow）**

  1. **ブランチ作成** — 作業開始前に `main` から feature ブランチを切る。ブランチ名は作業内容を端的に示す（例: `feature/riichi-detection`、`fix/undo-pass-bug`、`docs/ai-readme`）
     ```bash
     git switch main
     git pull
     git switch -c feature/<作業名>
     ```
  2. **コミット** — 小さい単位でこまめにコミットする。コミットメッセージは変更の「なぜ」を説明する
  3. **プルリクエスト** — 実装が完了したら `main` への PR を作成する。PR タイトルは 70 文字以内、本文に変更内容・テスト方法を記載する
  4. **マージ後の後始末** — マージ後はローカル・リモートともにブランチを削除する
     ```bash
     git switch main && git pull
     git branch -d feature/<作業名>
     ```

  **禁止事項**
  - `main` ブランチへの直接コミット（緊急の typo 修正など極めて軽微なものを除く）
  - `git push --force` を `main` ブランチに対して実行すること
  - **ユーザーの明示的な指示なしにコミットを実行すること** — 実装完了後はコミットメッセージ案を提示してユーザーの承認を得てからコミットする。「コミットしてください」「commit して」などの明示的な指示があった場合のみ実行する

  **コミットメッセージの書き方**
  - コミットメッセージは**日本語**で記述する
  - 型プレフィックス（`feat`/`fix`/`docs`/`refactor`/`test` など）は英語のままでよい
  - **メッセージは「何をした」よりも「なぜそうしたか」を説明する** — 変更の意図・理由・背景を含める
  - 例:
    - ✓ `fix: Undo 後に AI ターンが残る不具合を修正（スナップショット復元漏れでターンがずれるため）`
    - ✗ `fix: GameEngine.Undo() を修正`（何をしたかだけで理由がない）
  - 例: `feat: 四人麻雀にリーチ判定を追加`、`docs: AI 層の README を追加`

- **テスト駆動開発を意識すること**: 新機能追加・バグ修正の際は、Red → Green → Refactor のサイクルを厳守する。

  **Red → Green → Refactor サイクル**

  1. **TODO リスト作成（実装前）** — 実装に入る前に、何をテストするかをリストアップする。コーディング中は視野が狭くなるため、冷静な状態で考える。
  2. **テスト記述** — 1 サイクルで倒すテストは必ず 1 つ。複数テストを同時に通そうとしない。
  3. **Red 確認（必須）** — テストを実行して失敗することを確認する。「期待したエラーメッセージ」であることまで検証してから次に進む。この確認を怠るとバグ原因の特定が困難になる。
  4. **Green（最短経路で合格）** — まず動くコードを書く。一時的な汚さは許容し、リファクタリングは次のフェーズに回す。
  5. **Refactor（外部ふるまいを変えず内部品質を上げる）** — テストが Green の状態を保ちながら実施する。こまめにテストを実行し、Red になったら直前の変更を見直す。時間制限を設けて終わりを決める。
  6. **繰り返し** — TODO リストから完了項目を消し、次のテストに進む。

  **アンチパターン（やってはいけないこと）**
  - 複数テストを同時に倒そうとする
  - Red 確認をスキップして実装に入る
  - Refactor 中にテスト実行を怠る
  - Refactor に時間制限を設けず終わりのないリファクタリングを続ける

- **作業完了時の要件**: 作業を終了する際は、以下の状態で完了すること。これらの要件を満たさずに作業を終了しない
  1. **ビルドエラーがないこと** — 変更した範囲の `dotnet build` が成功すること
  2. **警告がないこと** — 未使用 using・未使用フィールドなど、IDE/コンパイラの警告が出ていないこと
  3. **テストが全てパスしていること** — `dotnet test` で全テストが成功すること
  4. **コミット・プッシュが完了していること** — ただし**ユーザーの明示的な指示がない限りコミットは実行しない**。コミット可能な状態まで整えたうえでコミットメッセージ案を提示する
- **コメント規約（XML doc）**: `public` / `internal` メンバーには `<summary>` を記載する。テストメソッドの `<summary>` には「何をすると何になるか（パス条件）」を明記する（例: `パス条件: IsSuccess = true かつ GameState が WhiteTurn に変わること`）
- **無関係な既存バグを独断で処理してしまわないこと**: 作業中に今のタスクと無関係な既存バグを見つけた場合、黙って直す・黙って無視するのではなく、見つけた時点でユーザーに扱い（今の PR に含めるか、別 PR に分けるか）を確認してから進めること
- **改行コードの混入確認漏れに注意**: ファイル編集後は `git diff --stat` で差分行数を確認すること。編集ツールが意図せず改行コードを変換すると、実際の変更が数行でもファイル全体が差分になってしまう。プロジェクト初期に `.gitattributes` を整備し、対象拡張子の改行コードを固定しておく

### テスト方針
- 単体テスト: 変更した関数・クラスの正常系・異常系・境界値を網羅する
- 結合テスト: 変更が影響する他機能（例: ゲームロジックの変更なら UI の挙動、AI との連携など）についても結合テストを追加し、機能間の連携が壊れていないことを確認する
- テストプロジェクトの構成（Othello_CSPY の `InternalsVisibleTo` 活用パターンなどを踏襲するか）は、ソリューション構成を決定するタイミングで別途プランニングする
